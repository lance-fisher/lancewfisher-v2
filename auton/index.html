What Auton Does

Concrete Improvements, Zero Surprises

Auton finds the small, safe improvements that accumulate into significant maintenance debt when ignored. Every proposal is scoped to a single file or concern, reviewed against project rules, and held for your approval.

Documentation Updates
COMMON
Finds outdated README sections, stale comments, and documentation that no longer matches the code. Proposes line-level corrections.
Dependency Checks
COMMON
Identifies outdated dependencies, deprecated APIs, and version mismatches. Proposes targeted upgrades with test verification.
Config Consistency
COMMON
Detects mismatches between environment files, build configs, and deployment settings. Proposes alignment without breaking changes.
Stale Reference Cleanup
COMMON
Finds dead imports, unused variables, broken internal links, and references to files or functions that no longer exist.
A Typical Cycle

How a Proposal Reaches You

Auton scans your workspace, finds outdated documentation in a project, proposes a fix, waits for your approval, applies it in a sandbox, runs tests, and only then writes to the real project tree. If anything fails, the change is rolled back automatically.

1
Scan
Auton walks every registered project, hashes file trees, and detects what has changed since its last pass.
2
Plan
A local LLM (running on your hardware, not a cloud API) reviews the changes and proposes small, bounded improvements.
3
Review
Each proposal is checked against project rules and the protected-path list. Anything out of scope is rejected automatically.
4
Your Approval
The proposal lands in your queue. You see exactly what will change, in which file, and why. You approve or reject it. No batch mode, no auto-apply.
5
Test and Apply
Approved changes run in a sandbox first. Tests must pass before the real project tree is touched. A failed test is an automatic rollback.

Worker Lifecycle

Pipeline

Every cycle moves through the same fixed stages. The Approval Gate is hard-coded between proposal and execution. There is no path that bypasses it. There is no "auto-apply" mode.

agents/

Agent Roles

Six specialized agents, each backed by a single Python module. Every agent inherits a common base, runs cooperatively on the EventBus, and writes its activity to the JSONL journal.

core/ · integrations/

Subsystems

Auton is organized into a small set of cooperating subsystems. The core is intentionally minimal: an event bus, persistent state, an append-only journal, a credential vault, and a safety layer that all writes flow through.

By Design

Safety Model

DRY_RUN is not a debugging convenience. It is the default running mode. Live execution is the explicit, deliberate exception, and even in live mode every individual change still passes through an approval gate.

DRY_RUN (default)
All proposals queued. Nothing applied. Journal records every intended change. Safe to leave running indefinitely.
EXECUTE (explicit)
Operator-flipped per session. Each individual proposal still requires an approval gate. No batch auto-apply.
core/safety.py

Approval Gates

Every change crosses these gates in order. A failure at any gate sends the proposal back to the journal as rejected and the cycle continues without applying anything.

Event Journal Schema

Every agent action, every proposal, every gate decision is appended to journal/journal.jsonl. Daily event logs are also rotated under logs/events_YYYY-MM-DD.jsonl. Both are append-only and together form the source of truth for what the worker has done.

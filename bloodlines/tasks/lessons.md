# Bloodlines Lessons

- 2026-04-13: Bloodlines has multiple visible roots, but the live implementation surface and newer continuity docs may have moved ahead of older archive status files. Re-verify the actual runtime and repo topology before treating any historical status document as current truth.

- 2026-04-14: Browser ES module caching is aggressive. A `window.location.reload()` or `?cacheBust=` query string on the HTML document does NOT invalidate the cached `./src/game/*.js` modules. To force fresh module loading after an edit, stop and restart the preview server. This saves minutes of debugging "my edit didn't land" confusion.

- 2026-04-14: The PreToolUse security hook in this workspace flags `innerHTML` assignment with template-literal interpolated content as a potential XSS vector, even when the data is wholly trusted (house names from JSON, faction display names, member titles from hardcoded templates). Rebuild new UI code using `document.createElement`, `textContent`, and `appendChild` rather than `innerHTML`. Existing innerHTML sites do not need to be rewritten; only new additions need the DOM-safe pattern.

- 2026-04-14: When refactoring damage pipelines, decouple the "reduce health" step from the "apply death effects" step. Store the killer faction id on the dying entity and let a finalization pass run at the end of the tick. This makes death consequences fire for any lethal path (damage, starvation, future disease) without threading attacker-context through every possible death source.

- 2026-04-14: For canonical design integrations that the operator explicitly marks non-negotiable, treat the design doctrine as a top-level canonical file in `01_CANON/`, then propagate integration entries additively across `CANONICAL_RULES.md`, `CANON_LOCK.md`, every relevant system file in `04_SYSTEMS/`, mechanics index, match flow, unit index, `docs/DEFINITIVE_DECISIONS_REGISTER.md`, `docs/BLOODLINES_CURRENT_STATE_ANALYSIS.md`, `docs/unity/PHASE_PLAN.md`, `docs/unity/SYSTEM_MAP.md`, the bible export (bump version with full section addition), and `00_ADMIN/CHANGE_LOG.md`. The doctrine must appear consistently across the canon surface so no future session can reduce it by reading only one file.

- 2026-04-14: When the operator asks for "the latest full bloodlines bible," the canonical export lives in `18_EXPORTS/`. The v3.2 file is the authoritative working version. To update without losing the prior version, append a new Part with the new session content, bump the internal version header, and create a new filename (v3.3) by copying. Keep the v3.2 file in place for backward-compatible references. Place the canonical desktop copy at `D:/Lance/Desktop/` (Windows Desktop redirected from C:/Users/lance/Desktop to D:/Lance/Desktop).

- 2026-04-14: Bible-builder recursion guard matters. `scripts/build_complete_bible.py` sweeps `18_EXPORTS/*.md` for Part S (Prior Exports). It excludes its own output via `EXCLUDE_FILENAMES` and `EXCLUDE_PATH_PATTERNS`, but a companion builder like `scripts/_build_everything_master.py` that also writes into `18_EXPORTS/` will be picked up and cause a ~2x size recursion on the next rebuild. Fix: add the companion output's filename prefix (e.g. `18_EXPORTS/BLOODLINES_COMPLETE_MASTER_`) to the exclude-path-patterns. Lesson: any new file written into `18_EXPORTS/` should be added to the builder's exclude patterns AT THE SAME TIME to prevent recursion on the next run.

- 2026-04-14: Coordinated cross-session builds. When two Claude sessions run in parallel on the same repo, neither should modify the other's files. The coordination pattern that worked: (a) other session owns `01_CANON/BLOODLINES_DESIGN_BIBLE.md`, `BLOODLINES_MASTER_MEMORY.md`, `BLOODLINES_APPEND_ONLY_LOG.md`, and the `scripts/build_complete_bible.py` generator; (b) this session owns the Dynasty Consequence Cascade runtime code, the Defensive Fortification Doctrine canon drop, and companion export scripts; (c) both sessions coexist additively via the `19_ARCHIVE/PRE_2026-04-14_CONSOLIDATION/` snapshots; (d) I ran the other session's build script unmodified (and later added a narrow exclude-pattern entry to prevent recursion, which is a purely-additive fix that does not reduce their coverage). The final everything-master reads the v2.0 output and extends it with code/data/tests/tasks/scripts/unity-shell. Both outputs live at the repo root-level desktop folder.
